SOLFAIL

Decode Solana transaction failures into clear explanations and actionable fixes.

Watch Demo Open Full README

Problem -> Solution

Solana errors are often accurate but still unusable in real product flows. A raw failure like {"InstructionError":[0,["Custom",6001]]} does not help a wallet, a dApp engineer, or an alerting pipeline understand what to do next.

Solfail turns those low-level failures into readable categories, plain-English explanations, and practical fixes that teams can pipe into UX, debugging, CI, and monitoring.

Quick Example

Input
{"InstructionError":[0,["Custom",6001]]}

Output
{
  "status": "FAILURE_DETECTED",
  "failureCategory": "anchor_constraint_owner",
  "confidence": "high",
  "explanation": "Anchor constraint violation: account owner does not match expected program.",
  "likelyFix": "Verify the account is owned by the correct program or initialize it properly."
}

CLI Demo

npx solfail decode tx.json --devnet

{
  "status": "FAILURE_DETECTED",
  "failureCategory": "missing_signer",
  "confidence": "high",
  "explanation": "A required signer account was not included when submitting the transaction.",
  "likelyFix": "Ensure the missing account signs the transaction before sending."
}

How It Works

Failed Transaction
↓
simulateTransaction RPC

Solfail Parser
↓
Error Mapping Engine
↓
Human Explanation + Fix

Why Solfail

  • Developer productivity through readable error diagnostics instead of raw RPC output.
  • Infra-grade decoding with stable categories that are safe to consume programmatically.
  • CLI + API + library support for different integration surfaces.
  • Multi-network support across devnet, testnet, and mainnet-beta workflows.

Use Cases

  • Wallet UX improvements
  • dApp debugging
  • CI/CD transaction validation
  • Blockchain monitoring
